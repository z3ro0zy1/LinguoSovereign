# 客观题性能改造说明

## 背景

LinguoSovereign 的客观题页面主要由 [ObjectiveRenderer.tsx](/Users/ronaldlee/Desktop/LinguoSovereign/src/components/eval/ObjectiveRenderer.tsx) 驱动。  
这个组件同时承担了：

- 计时器
- 听力音频播放
- 阅读材料渲染
- 题目渲染
- 划线高亮
- 草稿恢复
- 本地持久化
- 整卷提交

在功能持续叠加后，这个页面出现了明显的性能耦合问题。

## 改造前是什么样

### 1. 计时器放在顶层组件状态

改造前，`ObjectiveRenderer` 使用顶层 `timeSpent` 状态，并通过 `setInterval(..., 1000)` 每秒更新一次。

这意味着：

- 每一秒都会触发整个 `ObjectiveRenderer` 重渲染
- 左侧 passage / transcript 区域会跟着重渲染
- 右侧所有题卡也会跟着重渲染
- 听力过程中即使用户没有操作，整页也持续刷新

这不是“每秒 60 次”重渲染，而是“每秒 1 次整页重渲染”。  
频率不算夸张，但由于组件体量很大，这 1 次已经足够昂贵。

### 2. 草稿保存和计时绑定

改造前，草稿保存逻辑同时依赖：

- `answers`
- `timeSpent`

只要用户打字，或者秒表每走 1 秒，就会触发一次持久化。

### 3. `localStorage` 写入次数过多

改造前的 [testSession.ts](/Users/ronaldlee/Desktop/LinguoSovereign/src/lib/testSession.ts) 每次保存会拆成 4 次同步写入：

- `linguo_ans_*`
- `linguo_req_*`
- `linguo_cat_*`
- `linguo_time_*`

而 `localStorage` 是同步 API，会阻塞主线程。  
对于长篇阅读、听力和频繁输入场景，这种写法会明显增加卡顿概率。

### 4. 提交逻辑优先读本地缓存

改造前，整卷提交时主要从 `getUnitState()` 读取数据。  
如果用户刚输入完内容，还没来得及写入本地缓存，就存在“提交读到旧值”的风险。

## 改造后是什么样

### 1. 计时器改成局部组件

新增了 `ObjectiveTimerDisplay`，把秒表显示从顶层组件拆开。

现在的逻辑是：

- 计时器自己内部每秒刷新显示
- 父组件不再每秒 `setState`
- 当前秒数只同步到 `timeSpentRef`
- `timeSpentRef` 只作为“快照”使用，不驱动整页渲染

这样做之后：

- 秒表还在实时走
- 页面主体不再因为计时器而每秒全量刷新

### 2. 草稿保存改成“仅答案变化时”防抖保存

现在客观题页只在 `answers` 变化时触发持久化，不再因为秒表走动而写盘。

并且保存不是立即执行，而是做了短暂防抖：

- 答案变化
- 等待一小段时间
- 再把当前答案和当下时间快照一起写入

这样既保留了“刷新可恢复”的能力，又避免了无意义的高频同步 IO。

### 3. `testSession` 改成单 key JSON 存储

现在一个单元状态会统一存成一个对象：

- `answers`
- `reqIds`
- `category`
- `timeSpent`

只写一个 key，而不是 4 个 key。

同时保留了对旧 key 的兼容读取，所以不会导致已有本地草稿直接丢失。

### 4. 提交优先使用当前内存态

现在提交时，如果提交的是当前页面对应的 unit：

- 直接用内存中的最新答案
- 直接用 `timeSpentRef` 里的最新时间

只有其他关联 unit 才继续走 `getUnitState()`

这样可以避免“防抖保存尚未落盘，但用户已经点提交”的脏读问题。

### 5. 主观题同步使用同样的保存策略

[SubjectiveRenderer.tsx](/Users/ronaldlee/Desktop/LinguoSovereign/src/components/eval/SubjectiveRenderer.tsx) 也已经切换到：

- 只在答案变化时保存
- 使用防抖保存
- 当前页提交优先走内存中的最新答案

这样整个项目在持久化策略上保持一致，不再是不同题型各写一套高频写盘逻辑。

### 6. 题目 stem 解析下沉到单题组件

第二阶段又继续处理了一个更隐蔽但很贵的问题：

- 题干 HTML 的 `parse(...)`
- `{{response}}` 输入框替换
- 正误样式拼装

这些逻辑原本都发生在 `ObjectiveRenderer` 顶层的 `unit.questions.map(...)` 内部。  
只要父组件因为任意原因重渲染，所有题目的 stem 都要重新执行一遍 HTML 解析。

现在改成了：

- 抽出 `ObjectiveQuestionItem`
- 每一题内部自己 `useMemo` 解析题干
- 依赖只绑定到“当前题自己的答案、结果、字号和题干内容”

这意味着：

- 改第 3 题答案，不会再把第 1、2、4、5 题的 stem 全部重新 parse
- 听力和阅读长题组下，输入时的 CPU 压力会明显下降

### 7. Passage 区静态块改成缓存渲染

左侧 passage / transcript 区虽然还没有彻底拆成独立组件，但已经先做了第一层缓存：

- `unit.passage` 渲染结果改成 `useMemo`
- 文章内容只在题目数据本身变化时重新构建

这一步的目的不是“完全阻止左侧重渲染”，而是先避免在父组件重渲染时重复执行整段 passage block 组装。

### 8. 左侧材料区抽成独立 memo 组件

第三阶段继续往前走了一步，把左侧整块材料区正式抽成了 `ObjectivePassagePanel`。

这块区域里原本混着：

- passage / transcript 标题栏
- 听力音频播放器
- 阅读材料正文
- transcript 折叠状态
- 划线菜单
- 清除高亮菜单

这些内容虽然都属于“左侧阅读/听力材料”，但之前是直接写在 `ObjectiveRenderer` 主体里。  
这会导致一个问题：

- 右侧答案变化时，左侧材料区即使逻辑上不需要更新，也会一起进入 render 流程

抽成 `memo` 组件后，渲染边界更清晰了：

- 右侧答题
- 左侧材料

只有当左侧真正相关的 props 变化时，这一整块才会重新 render，例如：

- transcript 展开/折叠
- 材料内容变化
- 字号变化
- 高亮菜单状态变化

而普通的答题输入，不再默认带着左侧 passage/audio 一起走。

### 9. 结果查找从重复遍历改成 `Map`

之前右侧渲染每一道题时，都会执行一次：

- `result.results?.find(...)`

题目数量越多，这种重复线性查找越浪费。

现在改成：

- 先把结果数组一次性整理成 `Map<questionId, result>`
- 题目渲染时直接 `resultMap.get(q.id)`

这属于小优化，但在题数较多时是稳定收益。

### 10. 高亮交互从主组件里剥离

第四阶段处理的是左侧阅读区里最容易被忽视的一组状态：

- 当前选区 `selectionRange`
- 新划线菜单坐标
- 清除高亮菜单坐标
- 当前被点击的 `<mark>`
- 文档级点击收口事件
- 实际 DOM 包裹/拆除高亮逻辑

这些状态本身和答题结果没有直接关系，但以前都放在 `ObjectiveRenderer` 里。  
结果是：

- 主组件职责变重
- 高亮菜单开关也会让主组件承担更多交互状态
- 文档级事件订阅和 DOM 操作逻辑混在页面骨架里

现在改成了两层：

- `useObjectiveHighlighting`
  负责高亮状态、文档事件和 DOM 操作
- `ObjectiveHighlightMenus`
  负责菜单展示层

这样之后：

- `ObjectiveRenderer` 不再直接管理这组细粒度高亮状态
- 高亮能力变成可替换模块，而不是主页面里的内联细节
- 后面如果要继续做高亮持久化、句子定位、AI 解析联动，改动面会更小

这一步带来的提升更多是“稳定性和可维护性”，不是单纯追求某个毫秒级指标。  
但对性能也有帮助，因为页面主组件不再承载这组交互逻辑的实现细节。

### 11. 顶部工具栏从页面骨架中独立

第五阶段又继续收缩了 `ObjectiveRenderer` 的职责，把顶部工具栏抽成了 `ObjectiveToolbar`。

这一块原本同时负责：

- 计时器展示
- 字号调节
- 提交按钮
- 分数状态

虽然这一块本身不算很重，但它之前直接写在主组件 JSX 里。  
随着客观题页不断加功能，主组件会越来越像“所有东西都在这里”的容器。

抽出来之后的价值是：

- `ObjectiveRenderer` 更像页面编排层
- 工具栏的条件分支不再继续膨胀主组件
- 后面如果要继续加按钮、模式提示、诊断信息，可以在工具栏组件内部演进

这一步主要提升的是结构稳定性和未来的扩展成本。

### 12. 右侧题目区从页面骨架中独立

同一阶段还把右侧题目容器抽成了 `ObjectiveQuestionPanel`。

这块区域负责两件事：

- 题号导航 portal
- 真正的题目滚动区

以前这两部分和主页面布局混在一起，现在独立后：

- 右侧题目区有了清晰边界
- `ObjectiveRenderer` 不再直接持有题目区 DOM 骨架
- 后面如果要继续做：
  - 题号导航增强
  - 题目虚拟滚动
  - 分组显示
  - 题型专属面板
  都可以在这个边界内继续扩展

这一阶段不是单纯追求“再少几次 render”，而是把剩余的大块结构进一步从主组件里拿出去，降低后续迭代成本。

## 为什么这样改

### 计时器本质上不应该驱动整页重渲染

秒表显示是高频 UI，但不属于“页面结构状态”。  
它应该局部刷新，而不是带着整篇阅读材料和整组题卡一起刷新。

### `localStorage` 不适合高频同步写入

`localStorage` 是同步 API。  
把它放在：

- 每秒计时
- 每次输入字符
- 多 key 连续写入

这种路径里，本质上就是在主线程里持续做同步 IO。

### 草稿恢复的目标不是“时间绝对精确”

草稿系统真正要保证的是：

- 答案别丢
- 页面刷新后能大致恢复进度

没有必要为了“时间秒级精确”去每秒写盘。

### 当前页提交必须以当前内存状态为准

只要引入防抖保存，就必须同时修正提交逻辑。  
否则优化了写盘频率，却把“最新输入未提交”变成新的正确性问题。

## 带来的提升

### 性能提升

- 客观题页不再因为计时器每秒整页重渲染
- 主线程同步写盘次数显著下降
- 用户在听力/阅读过程中输入答案时更稳定

### 稳定性提升

- 当前页提交不再依赖是否刚好已经写盘
- 本地存储结构更集中，更容易维护
- 对旧缓存格式保持兼容，升级成本低

### 结构上的提升

- 计时器逻辑和页面主体分离
- 持久化策略统一到“答案变化驱动”
- 单题渲染从“父组件统一 parse”改成“每题局部 memo parse”
- 左侧材料区从“内联渲染”改成“独立 memo 边界”
- 高亮系统从“页面内联逻辑”改成“hook + 展示组件”
- 顶部工具栏从“主组件内联 JSX”改成“独立组件”
- 右侧题目区从“主组件内联 JSX”改成“独立组件”
- 结果检索从重复遍历改成一次建索引
- 为后续继续拆 `ObjectiveRenderer` 打下基础

## 当前还没有解决的部分

这一轮主要解决的是：

- 高频重渲染
- 高频写盘
- 顶层组件里重复执行的 HTML parse
- 左右主区域没有独立渲染边界
- 高亮状态和文档订阅与页面骨架高度耦合
- 工具栏和题目区仍然膨胀主组件结构
- 结果数组的重复线性查找

仍然存在的后续优化空间有：

- `ObjectiveRenderer` 仍然是大组件，还可以继续拆出：
  - 答题状态和提交流程控制层
- 高亮菜单与 DOM 操作虽然已经抽离，但高亮本身仍是 DOM 直改模式
- 高亮系统仍是 DOM 直改模式，后续可以继续做状态隔离与持久化
- 页面宽度/分栏逻辑仍在主组件内部
- 首页和详解页也还有进一步的拆分空间
