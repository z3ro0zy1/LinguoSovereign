# LinguoSovereign 口语模块全流程说明

本文档说明当前仓库里口语模块的完整设计、两条主要链路、关键文件、模型分工，以及 Gemini Live / ephemeral token 等知识点。

最后更新：`2026-04-11`

## 1. 口语模块现在分成哪两种模式

当前口语模块只保留两种训练模式：

1. `转录评分`
2. `自由对话`

它们的目标不同，所以底层模型和交互链路也不同。

### 1.1 转录评分

目的：
- 用户按题目一题一题作答
- 系统先拿到文本 transcript
- 最后把整套回答合并提交
- 由 AI 按 IELTS 口语维度评分

特点：
- 非实时
- 更像“训练后总结”
- 重点是 transcript 和评分结果

### 1.2 自由对话

目的：
- 用户和 AI 连续自然对话
- 尽量接近原生语音助手体验
- 中英双语自由切换

特点：
- 实时
- 原生音频输入输出
- 同步获得输入/输出转录
- 停止对话后不清空页面里的当前记录

---

## 2. 为什么要拆成两条链路

原因很简单：实时语音和非实时评分，不是同一个问题。

### 2.1 转录评分关注的是

- 低成本
- 稳定输出结构化评分
- 方便做总结、分维度反馈、训练建议

这类任务更适合：
- `Kimi 2.5`
- 或 `gemini-3.1-flash-lite-preview`

### 2.2 自由对话关注的是

- 低延迟
- 原生语音自然度
- 说完就接话
- 输入输出都是音频而不是“先文字再朗读”

这类任务不能用 `gemini-3.1-flash-lite-preview`，因为它官方能力明确写着：
- `Audio generation: Not supported`
- `Live API: Not supported`

所以自由对话必须用 `Gemini Live` 系列模型。

当前接法使用：
- `gemini-3.1-flash-live-preview`

## 4. Prompt 系统现在怎么工作

Prompt 不再只按大类 `Speaking` / `Writing` 区分，而是按“题型 + 用途”区分。

当前 `UserPrompt` 表支持这三类典型场景：

- `Writing + evaluation`
- `Speaking + transcript_eval`
- `Speaking + free_chat`

### 4.1 为什么不新建表

因为原来的 `UserPrompt` 已经够用了，问题不在“能不能存”，而在“能不能区分用途”。

新增 `purpose` 字段就够：

- `evaluation`
- `transcript_eval`
- `free_chat`

### 4.2 Prompt 选择优先级

当前系统统一遵循这个优先级：

1. 前端当前编辑的 `promptContent`
2. 指定 `promptId`
3. 数据库中 `isDefault = true` 的默认 prompt
4. 代码中的 fallback prompt

这个规则适用于：
- transcript 评分
- free chat

## 5. 转录评分模式的完整流程

### 5.1 用户看到什么

在 Speaking 的 `转录评分` 模式里：

- 左侧是题目列表
- 当前题高亮
- 用户一题一题答
- 右侧是 transcript 输入区
- 最后统一提交评分

### 5.2 前端流程

前端文件：
- [src/components/eval/SubjectiveRenderer.tsx](/Users/ronaldlee/Desktop/LinguoSovereign/src/components/eval/SubjectiveRenderer.tsx)

主要逻辑：

1. 加载 speaking 题组
2. 用户逐题录入或修订 transcript
3. 所有答案保存在本地状态
4. 点击最终评分时，把整套 transcript 一次性发到后端

### 5.3 后端流程

后端文件：
- [src/app/api/eval/subjective/route.ts](/Users/ronaldlee/Desktop/LinguoSovereign/src/app/api/eval/subjective/route.ts)

主要逻辑：

1. 接收 speaking transcript
2. 拉取当前场景 prompt
3. 拼接评分用 system prompt
4. 调用 OpenAI-compatible 客户端
5. 当前 speaking transcript 评分可接 `Kimi 2.5`
6. 要求模型返回结构化 JSON
7. 存入 `Submission`

### 5.4 为什么 transcript 评分不走 Live

因为 transcript 评分本质上不是“实时语音对话”，而是“文本分析 + 结构化反馈”。

用 Live 模型做这件事：
- 成本更高
- 控制更差
- 没必要

## 6. 自由对话模式的完整流程

### 6.1 旧方案为什么不够好

旧方案是：

1. 浏览器 `SpeechRecognition`
2. 后端返回文本
3. 浏览器 `speechSynthesis` 朗读文本

这个方案的问题：

- 输入转录依赖浏览器，自然度和多语言稳定性有限
- 输出不是模型原生语音，而是浏览器本地朗读
- 中文、英文切换很容易不自然
- 声音不像 ChatGPT / Gemini App 的原生 voice

### 6.2 新方案是什么

新方案是：

1. 前端向我方后端请求 Live token
2. 后端签发 Gemini Live ephemeral token
3. 浏览器用 token 直连 Gemini Live
4. 麦克风采集原始音频
5. 音频转成 `16kHz PCM16`
6. 通过 Live WebSocket 实时送入模型
7. 模型返回：
   - 原生音频块
   - 用户输入转录
   - 模型输出转录
8. 前端把模型音频排队播放，同时把转录显示在对话记录里

### 6.3 为什么要用 ephemeral token

因为浏览器不能直接拿长期 API key 连供应商。

如果把长期 `GEMINI_API_KEY` 暴露到前端：

- 安全性差
- 容易泄漏
- 会被随意滥用

所以当前做法是：

1. 服务端持有长期 API key
2. 服务端创建一个短期 token
3. 短期 token 只允许 Live 会话使用
4. 浏览器只拿这个短期 token 去连 Gemini Live

这就是 `ephemeral token` 的意义。

### 6.4 `/api/speaking/live` 现在具体做什么

文件：
- [src/app/api/speaking/live/route.ts](/Users/ronaldlee/Desktop/LinguoSovereign/src/app/api/speaking/live/route.ts)

它现在只做两件事：

1. 鉴权
2. 签发 Live token

流程：

1. 校验登录状态
2. 读取当前题组
3. 读取用户当前场景 prompt
4. 把题目 bundle 和 prompt 合成 system instruction
5. 调用 `createGeminiLiveToken()`
6. 返回：
   - token
   - model
   - unitTitle
   - questionBundle

### 6.5 `createGeminiLiveToken()` 做了什么

文件：
- [src/lib/ai.ts](/Users/ronaldlee/Desktop/LinguoSovereign/src/lib/ai.ts)

它的职责：

1. 用服务端 API key 初始化 `GoogleGenAI`
2. 使用 `v1alpha`
3. 创建 Live token
4. 把关键 Live 配置锁进 token

锁进去的配置包括：

- `model`
- `responseModalities = [AUDIO]`
- `systemInstruction`
- `inputAudioTranscription`
- `outputAudioTranscription`
- `realtimeInputConfig`

### 6.6 为什么要锁定这些字段

原因是前端不应该随意改 Live 会话的关键参数。

否则容易出现：

- 前端把音频响应改成文本响应
- 忘记开输入转录
- 忘记开输出转录
- 同一页面上不同用户乱改系统提示词结构

所以 token 不只是“通行证”，也是“能力边界和会话配置的约束”。

### 6.7 `uses` 是什么

`uses` 表示这个 token 允许被创建多少次 Live session。

当前默认值：
- `GEMINI_LIVE_TOKEN_USES`
- 默认 `5`

为什么不是 `1`：

因为当前交互要求是：
- 用户点击“停止对话”后不清空内容
- 同一页面里还可以再继续

如果 `uses = 1`，停止后再开就得重新签 token。
可以这么做，但前端交互会更频繁触发签发流程。

当前设置成可复用几次，页面体验更平滑。

### 6.8 `newSessionExpireTime` 和 `expireTime` 有什么区别

- `newSessionExpireTime`
  - 到这个时间后，不能再用这个 token 新开会话

- `expireTime`
  - 到这个时间后，连已建立的会话也可能被拒绝或提前关闭

当前实现里：

- 新会话窗口：约 5 分钟
- token 总寿命：约 30 分钟

这是比较稳妥的本地开发默认值。

### 6.9 Live 音频格式知识点

Gemini Live 的音频不是现成的 mp3 / wav，而是原始 PCM 数据。

输入：
- `16kHz`
- `16-bit PCM`
- `little-endian`

输出：
- `24kHz`
- `16-bit PCM`

所以前端必须做两件事：

1. 录音重采样
2. 模型音频解码播放

### 6.10 前端麦克风为什么要重采样

浏览器麦克风一般给你的是：

- `44.1kHz`
- 或 `48kHz`

但 Live API 最适配的是 `16kHz PCM16` 输入。

所以 [src/components/eval/SpeakingAiRenderer.tsx](/Users/ronaldlee/Desktop/LinguoSovereign/src/components/eval/SpeakingAiRenderer.tsx) 里做了：

1. 从 `AudioContext` 里拿原始浮点采样
2. 重采样到 `16kHz`
3. 转成 `Int16`
4. base64 编码
5. `session.sendRealtimeInput()`

### 6.11 模型返回的音频为什么还要“排队播放”

因为 Live 返回的是一段一段音频块。

如果每块都立刻播放，可能出现：

- 抢播放头
- 乱序
- 断裂感

所以前端维护了一个 `playbackCursor`：

1. 每收到一块音频
2. 转成 `AudioBuffer`
3. 计算它应该在时间轴上的开始时间
4. 顺序排到同一个 `AudioContext` 里播放

这样语音会连续很多。

### 6.12 为什么模型说话时要关麦

如果模型输出音频时麦克风还一直开着，就容易出现：

- 扬声器回声被再次录进去
- 模型“听到自己”
- 触发错误转录

所以当前实现里：

- 一旦收到模型输出转录或音频块
- 会先停止麦克风采集
- 等模型播放结束后，再自动恢复开麦

### 6.13 停止对话为什么不清空记录

因为用户明确要求：
- 只有退出页面时自然失去当前会话
- 停止对话只是停会话，不是删历史

所以现在 `stopConversation()` 做的事是：

- 关闭 Live session
- 关闭麦克风
- 清空播放队列状态

但不会清：
- `conversation`

### 6.14 重新开始为什么还能“继续聊”

因为重新建立 Live session 后，前端会把当前页面上的对话历史作为初始上下文重新发给模型。

也就是：

1. 旧会话停掉
2. 新会话建起来
3. 把页面里的 `conversation` 映射成：
   - `user`
   - `model`
4. 调 `sendClientContent()`
5. 然后再恢复麦克风

因此用户感受到的是“同一轮对话继续”，而不是重新从零开始。

## 7. 为什么 Gemini App 听起来比旧网页方案自然

不是 prompt 的问题，核心是语音栈不同。

### 7.1 旧网页方案

- 浏览器转文字
- 模型回文字
- 浏览器再把文字朗读出来

这是“文本中转语音”。

### 7.2 Gemini Live / 原生语音方案

- 用户直接给音频
- 模型直接回原生音频
- 中间不是浏览器 TTS

这是“音频到音频”的原生会话。

所以自然度更高。

## 9. 现在有哪些已知边界

### 9.1 页面刷新后，对话不持久化

当前 `free chat` 的 `conversation` 只保存在组件状态里。

这意味着：

- 页面内停止再继续：可以
- 刷新页面后恢复：不行

如果要做持久化，需要新增表，例如：

- `SpeakingSession`
- `SpeakingTurn`

### 9.2 transcript 评分和 free chat 仍然是两条链路

当前不会自动把自由对话记录拿去生成 IELTS 评分。

如果后面要做：
- 对话结束后一键评分
- 把 free chat transcript 自动沉淀为 submission

需要再补一层汇总逻辑。

### 9.3 当前没有保存原始音频文件

现在：

- free chat 用的是实时音频流
- 但没有把每轮音频落盘

如果后面要做：
- 回放用户原声
- 发音级别分析
- 逐句精听复盘

就需要接文件存储。

## 10. 一句话总结当前口语架构

当前口语模块已经明确拆成两条链路：

- `转录评分`
  - 走 transcript
  - 适合 Kimi 2.5 / Flash-Lite
  - 负责 IELTS 评分与反馈

- `自由对话`
  - 走 Gemini Live 原生音频
  - 服务端签发 ephemeral token
  - 浏览器直连 Live API
  - 负责自然实时语音交流

这套拆法的核心价值是：

- 实时体验和评分能力分开
- 模型选型更准确
- 成本更可控
- 以后要做录音持久化、对话后评分、复盘页扩展，都有清晰落点
