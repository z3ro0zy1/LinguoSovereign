# 项目总结

## 1. 项目一句话定位

`LinguoSovereign` 是一个基于 `Next.js 16 + React 19 + TypeScript + Prisma + PostgreSQL + NextAuth + OpenAI` 实现的 IELTS 练习平台，覆盖阅读、听力、写作、口语、AI 评分、复盘和数据统计。

它不是“只把页面做出来”的 Demo，而是一个已经具备完整业务闭环的产品原型：

- 题库管理
- 用户登录注册
- 做题流程
- 本地进度保存
- 提交入库
- AI 评分
- 历史复盘
- 数据看板

## 2. 我给这个项目的分档

### 综合分档：`中上档 / 准生产级作品`

如果把学生项目按面试价值粗分成 4 档：

| 档位 | 特征 |
| --- | --- |
| 1 档 | 静态页面 / 纯前端练手 |
| 2 档 | 有前后端联调，但业务链条不完整 |
| 3 档 | 有完整业务闭环、鉴权、数据库、状态持久化、可部署 |
| 4 档 | 架构、性能、测试、监控、边界处理都比较成熟，接近正式商业化 |

这个项目我会放在 **3 档偏上**。

### 为什么不是低档

- 用了真实的 `App Router + Route Handlers + Prisma + PostgreSQL`
- 有完整的“题目 -> 作答 -> 提交 -> 评分 -> 复盘 -> 统计”链路
- 有登录鉴权和受保护页面
- 有本地草稿恢复和连考流程
- 写作/口语引入了 AI 评分与对话能力，不是普通 CRUD

### 为什么还没到更高档

- 目前测试体系基本缺失
- API 校验主要靠手写逻辑，缺少 `zod` 这类统一 schema 校验
- 数据模型还比较轻，题目、提交、评分、提示词等实体可以继续拆分
- 错误处理、可观测性、缓存策略、权限粒度还不算完整
- TypeScript 虽然开启了 `strict`，但局部仍有 `any`，类型边界没有完全收紧

## 3. 这个项目是怎么落地实现的

## 3.1 技术架构

从代码看，这个项目采用的是比较典型的 **Next.js 全栈一体化架构**：

- 前端 UI：React Client Components + Tailwind + shadcn/ui
- 服务端页面：Next.js App Router 的 Server Components
- 后端接口：`src/app/api/**/route.ts`
- 数据访问：Prisma
- 数据库：PostgreSQL
- 登录鉴权：NextAuth Credentials + Prisma Adapter
- AI 能力：OpenAI SDK
- 本地草稿：`localStorage`

也就是说，它不是前后端完全分离，而是用 Next.js 把页面、接口、鉴权、数据库访问都放在一个项目里。

## 3.2 核心业务闭环

这个项目最值得讲的，是它的业务闭环完整。

### 第一步：首页拉取题库

首页在服务端直接查数据库：

- `src/app/page.tsx`

这里通过 Prisma 查询 `questionUnit`，然后把结果交给客户端组件 `DashboardClient` 展示。

这说明它的首页不是 mock 数据，而是真正从数据库拉题组列表。

### 第二步：用户进入做题页

做题页入口：

- `src/app/eval/[id]/page.tsx`

这里做了几件关键的事：

- 按 `id` 查询当前题组和题目列表
- 判断题型是否需要登录
- 写作、口语未登录会重定向到登录页
- 自动计算同一套题的 siblings，形成连考流程
- 把题目数据传给 `EvalWrapper`

这一步已经体现出比较成熟的业务意识：不是只打开一道题，而是支持整套 Cambridge Test 的连续练习。

### 第三步：EvalWrapper 分发不同题型

- `src/components/eval/EvalWrapper.tsx`

这里相当于题型路由层：

- 阅读/听力走 `ObjectiveRenderer`
- 写作/口语走 `SubjectiveRenderer`
- 统一管理提交结果
- 统一处理底部“继续下一模块”的导航逻辑

这是一种比较合理的组件分层方式：把“通用流程壳子”和“具体题型实现”分开。

### 第四步：客观题如何实现

- `src/components/eval/ObjectiveRenderer.tsx`
- `src/app/api/eval/objective/route.ts`

客观题链路很完整：

1. 前端渲染阅读材料 / 听力原文、题卡、题号导航
2. 记录答案、计时、音频播放状态
3. 草稿和耗时实时存到 `localStorage`
4. 提交时请求 `/api/eval/objective`
5. 服务端从数据库取标准答案
6. 执行答案比对、统计正确数
7. 将提交记录和评分结果写入 `Submission`
8. 返回逐题结果给前端展示解析

这里的落地点非常清晰：**标准答案永远以数据库为准，评分逻辑放在服务端而不是前端**。这在面试里是一个加分点，因为它避免了前端泄露完整判题逻辑。

### 第五步：主观题如何实现

- `src/components/eval/SubjectiveRenderer.tsx`
- `src/app/api/eval/subjective/route.ts`

主观题分两种模式：

- `Save Only`：只保存答案，不做 AI 判分
- `AI Evaluate`：保存答案并调用大模型评分

服务端流程是：

1. 查询题目 prompt
2. 读取默认 prompt 或用户自定义 prompt
3. 拼接 task context
4. 调用 OpenAI Chat Completions
5. 强制要求返回 JSON
6. 将维度分和反馈文本入库
7. 返回结构化评分结果

这套设计非常适合面试描述为：

> “我把主观题的 AI 评分做成了一个可配置的服务端评估流水线，前端只负责收集答案，服务端负责 prompt 组装、模型调用、结果解析与持久化。”

### 第六步：口语 AI 模式如何实现

- `src/components/eval/SpeakingAiRenderer.tsx`
- `src/app/api/speaking/live/route.ts`

这是项目里最像“亮点功能”的部分。

前端：

- 用浏览器 `SpeechRecognition / webkitSpeechRecognition` 做语音转文字
- 用 `speechSynthesis` 做文本转语音
- 维护 conversation 历史
- 每轮把用户话术发给后端

后端：

- 校验登录状态
- 根据当前 speaking unit 取题面
- 拼接系统 prompt + 历史对话 + 用户最新输入
- 调用模型生成考官回复
- 把回复返回前端继续语音播报

本质上这是一个简化版实时口语考官系统，虽然还是基于轮询式 HTTP，而不是 WebRTC，但对于学生项目来说已经很能说明问题。

### 第七步：复盘页如何实现

- `src/app/review/[id]/page.tsx`

复盘页分两种模式：

- 不带 `submissionId`：参考答案页
- 带 `submissionId`：历史作答复盘页

服务端负责：

- 校验用户身份
- 查询题组和提交记录
- 计算客观题/主观题的最终 band score
- 把结果交给 `ReviewClient`

这个设计很好，因为它把“纯题目资料页”和“个人作答复盘页”统一到了一个路由里，降低了页面分裂度。

### 第八步：统计页如何实现

- `src/app/api/analytics/route.ts`
- `src/app/dashboard/analytics/page.tsx`

统计页不是静态拼图，而是基于用户提交记录动态聚合：

- 读取用户全部 submissions
- 把 `Reading/Listening` 再按标题拆成 `Reading` 或 `Listening`
- 根据 `aiScore` 计算 band 分
- 聚合平均分、时间轴、历史记录
- 前端用 `recharts` 渲染趋势图

这里可以体现你的数据思维：不仅做了“提交”，还做了“提交后的可视化利用”。

## 4. 数据模型是怎么支撑业务的

- `prisma/schema.prisma`

核心表设计如下：

- `QuestionUnit`：一套题 / 一个模块
- `Question`：具体题目
- `Submission`：用户一次提交
- `UserPrompt`：AI 评分提示词
- `User / Session / Account / VerificationToken`：NextAuth 相关用户体系

这套模型的优点：

- 题库和用户提交分离
- 客观题、主观题都能统一落到 `Submission`
- `aiScore` 和 `aiFeedback` 用 JSON / 文本承接模型输出，灵活度高

它的不足也要会讲：

- `Question.type`、`QuestionUnit.category` 目前更多靠字符串驱动，后续可以枚举化
- `Submission.aiScore` 用 JSON 很灵活，但对复杂分析查询不够友好
- 还没有把 speaking live 对话历史单独建表

## 5. 用了 React 的哪些特性

这个项目不是只写 JSX，而是比较完整地用了 React 的组件化和状态模型。

### 5.1 Client Components

大量交互页面用了 `"use client"`：

- `DashboardClient`
- `AnalyticsDashboard`
- `EvalWrapper`
- `ObjectiveRenderer`
- `SubjectiveRenderer`
- `SpeakingAiRenderer`

原因很明确：这些页面依赖浏览器能力，例如：

- `useState`
- `useEffect`
- `useMemo`
- `useRef`
- `localStorage`
- `SpeechRecognition`
- `speechSynthesis`
- DOM selection / audio control

### 5.2 Hooks 驱动交互状态

项目大量使用了：

- `useState` 管理答案、录音状态、弹窗、加载态、语言状态
- `useEffect` 处理副作用，如计时器、自动播放音频、恢复草稿、拉取数据
- `useMemo` 做派生值缓存，如本地化模块配置、当前题面、context value
- `useRef` 持有音频、语音识别实例、对话历史、DOM 引用
- `useContext` + `createContext` 实现语言上下文

这说明项目作者不是把 React 当模板引擎，而是把它当状态管理与交互编排工具在用。

### 5.3 组件分层

这个项目的 React 组件分层比较清晰：

- 页面壳组件：`EvalWrapper`
- 业务组件：`ObjectiveRenderer`、`SubjectiveRenderer`
- 功能子组件：`ObjectiveQuestionCard`、`RestoreDraftDialog`、`QuestionNavigatorPortal`
- 基础 UI 组件：`button`、`dialog`、`select`、`tabs` 等

这种分层在面试里很好讲，因为它体现了“页面编排”和“功能实现”被拆开了。

### 5.4 Context 做国际化

- `src/components/LocaleProvider.tsx`
- `src/lib/i18n.ts`

这里用 `Context` 承担全局语言状态：

- `locale`
- `setLocale`
- `toggleLocale`
- `t(key)`

这是一种轻量但有效的 React 全局状态方案，适合中小项目。

### 5.5 Portal 思想

从 `QuestionNavigatorPortal` 的命名和 `footer-left-slot` 挂载点可以看出，项目使用了 portal 把题号导航投射到底部固定工具栏。这是比较“React 化”的实现，不需要把底部导航的结构硬塞进同一层 DOM。

### 5.6 浏览器能力与 React 结合

项目把 React 与浏览器原生 API 结合得比较多：

- 语音识别
- 语音播报
- 音频播放器
- 文本选区 Range
- `localStorage`
- `sessionStorage`

这类项目在面试中通常比普通表单项目更有辨识度。

## 6. 用了 Next.js 的哪些特性

这个项目最核心的框架价值其实就是 Next.js。

### 6.1 App Router

项目使用了 `src/app` 目录结构，说明是 App Router：

- `src/app/page.tsx`
- `src/app/eval/[id]/page.tsx`
- `src/app/review/[id]/page.tsx`
- `src/app/api/**/route.ts`

这意味着：

- 页面天然按路由组织
- 支持 Server Components
- 动态路由更自然
- 页面与接口统一在一个工程中

### 6.2 Server Components

像下面这些页面是服务端组件风格：

- 首页 `page.tsx`
- 做题页 `eval/[id]/page.tsx`
- 复盘页 `review/[id]/page.tsx`

这些页面直接在服务端：

- 查 Prisma
- 调 `getServerSession`
- `redirect`
- `notFound`

这是 Next.js 相比传统 React SPA 的一个核心提升点：**页面首屏数据获取和权限控制前置到服务端**。

### 6.3 Route Handlers

项目的后端接口全部使用 App Router 下的 Route Handlers：

- `/api/eval/objective`
- `/api/eval/subjective`
- `/api/speaking/live`
- `/api/analytics`
- `/api/register`
- `/api/profile`

这比单独再开一个 Express/Nest 服务更轻量，也更适合中小型全栈项目。

### 6.4 动态路由

项目大量用了动态路由：

- `eval/[id]`
- `review/[id]`
- `api/units/[id]`
- `api/auth/[...nextauth]`

这体现了 Next.js 的文件系统路由能力。

### 6.5 服务端跳转与异常边界

在页面组件里用到了：

- `redirect`
- `notFound`

这是 App Router 非常常见也非常实用的能力，特别适合做鉴权页和详情页。

### 6.6 Metadata 与 next/font

- `src/app/layout.tsx`

这里使用了：

- `export const metadata`
- `next/font/google`

说明项目已经考虑到：

- SEO / 页面标题
- 全局字体加载优化

虽然不是这个项目的核心卖点，但能说明作者对 Next.js 基础能力比较熟。

### 6.7 next-auth 与 App Router 融合

项目通过：

- `getServerSession(authOptions)`
- `SessionProvider`
- `useSession`

把服务端权限控制和客户端会话读取结合了起来。这是 Next.js 全栈项目里非常常见的一种成熟实践。

## 7. 用了 TypeScript 的哪些特性

项目的 TypeScript 使用水平是“够用且偏实战”的。

### 7.1 `strict: true`

- `tsconfig.json`

项目开启了 `strict`，这是一个很关键的信号，说明不是把 TS 当摆设。

### 7.2 类型别名与结构建模

项目中定义了不少业务类型：

- `AnalyticsRecord`
- `TimelinePoint`
- `AnalyticsData`
- `SubjectiveEvaluation`
- `StoredUnitState`
- `Locale`
- `I18nKey`

这说明作者在用 TS 描述业务对象，而不是只给 props 补最基础的类型。

### 7.3 联合类型

用了很多联合类型来表达有限状态：

- `type Locale = "zh" | "en"`
- `type DashboardTab = "home" | "Reading" | "Listening" | "Writing" | "Speaking" | "FullTest"`
- `mode?: "standard" | "ai"`
- `role: "assistant" | "user"`

这对前端项目很重要，因为它能减少状态字符串写错的问题。

### 7.4 工具类型

项目用到了 `Exclude`：

- `type ModuleTab = Exclude<DashboardTab, "home" | "FullTest">`

这说明作者不是只会写 interface，也会用 TypeScript 提供的内置工具类型。

### 7.5 泛型

例如：

- `safeParse<T>(value, fallback): T`
- `createContext<LocaleContextValue | null>(null)`
- `useMemo<LocaleContextValue>(...)`

这体现出对泛型的基本掌握。

### 7.6 类型断言与运行时防守

项目中有不少：

- `as`
- `typeof`
- `Array.isArray`
- `"id" in session.user`

这类写法说明作者知道：TypeScript 只在编译期有效，运行时还得自己做保护。

### 7.7 浏览器 API 的类型补充

在语音识别部分，手工声明了：

- `BrowserSpeechRecognition`
- `SpeechRecognitionEventLike`
- `declare global { interface Window { ... } }`

这是一个明显的实战点，因为很多浏览器 API 默认类型并不完善，必须手补。

### 7.8 还可以更好的地方

虽然用了 TS，但还不算非常硬核：

- 局部有 `any`
- API 请求体 / 响应体缺少统一 schema
- Prisma 返回值到前端 props 的映射还可以更强类型化
- 客观题与主观题共享接口的泛型抽象还不够

所以面试里更适合说“TS 用于业务建模和边界约束”，不要吹成“类型体系统一得非常彻底”。

## 8. 这个项目最值得你拿去面试讲的亮点

建议你重点讲下面 5 个点：

### 亮点 1：全栈一体化闭环

不是单纯前端页面，而是从数据库、鉴权、接口、页面、AI 评分到统计面板都打通了。

### 亮点 2：题型分层设计

通过 `EvalWrapper -> ObjectiveRenderer / SubjectiveRenderer`，把题型差异和公共流程拆开，代码组织清晰。

### 亮点 3：主观题 AI 评分流水线

服务端统一拼 prompt、调模型、解析 JSON、持久化评分结果，这个非常适合讲“AI 功能如何工程化落地”。

### 亮点 4：口语 AI 对话体验

把浏览器语音识别、语音播报和 LLM 追问串起来，做出了有交互感的产品体验。

### 亮点 5：本地草稿恢复

用 `localStorage` + 做题状态模型实现恢复进度，这种细节很能体现产品思维，不只是“功能能跑”。

## 9. 面试时可能被追问的问题

### 9.1 为什么客观题评分放服务端？

可以回答：

> 因为标准答案和评分逻辑放在前端容易被看见，也不利于统一记分和记录提交历史，所以我把判题逻辑放到服务端，并把前端当成纯输入层。

### 9.2 为什么主观题评分要强制 JSON 输出？

可以回答：

> 因为 AI 返回自然语言不稳定，不利于前端展示和数据库存储，所以我要求模型固定返回 JSON，方便我解析总分、维度分和反馈摘要。

### 9.3 为什么要用 localStorage 保存草稿？

可以回答：

> 做题场景很怕刷新丢失，我先用浏览器本地存储做低成本恢复机制，既不用频繁打服务端草稿接口，也能提升用户体验。

### 9.4 为什么首页用服务端查 Prisma，而统计页走客户端 fetch？

可以回答：

> 首页是首屏核心内容，适合服务端直接查库输出；统计页依赖用户登录态并且交互更强，我让客户端在登录确认后再请求接口，避免无意义渲染。

### 9.5 如果继续优化，你会怎么做？

建议回答这几个方向：

- 引入 `zod` 做 API schema 校验
- 把题型和 category 改成枚举
- 给 AI 评分结果设计更清晰的数据结构
- 增加单元测试和接口测试
- 给 speaking live 做更实时的流式交互
- 增加缓存、日志和监控

## 10. 如果把它包装成你的面试表述

你可以这样讲：

> 我做了一个基于 Next.js 的 IELTS AI 练习平台。项目采用 App Router，全栈放在同一个工程里，服务端用 Prisma 连接 PostgreSQL，鉴权用 NextAuth。系统支持阅读、听力、写作和口语四类题型，其中客观题走服务端判题，主观题走 OpenAI 评分流水线，口语还实现了浏览器语音识别 + AI 考官追问。为了让练习流程更接近真实产品，我还做了本地草稿恢复、历史复盘和统计分析面板。这个项目让我比较系统地把 React 交互、Next.js 全栈能力、TypeScript 类型建模和 AI 功能落地串了起来。 

## 11. 结论

如果你的目标是“拿这个项目当面试资料”，这个项目是够格的，而且比普通管理后台或纯 CRUD 项目更有辨识度。

它最适合你拿来证明三件事：

- 你不只会写页面，也能做全栈闭环
